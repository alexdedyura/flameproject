# Кодировка cp1251 (критично)

Исходные файлы с русским текстом **должны храниться в Windows-1251 (cp1251)**, и MySQL настроен
общаться в cp1251 (см. `core/mysql.inc`). Русский текст часто выглядит как «кракозябры» в UTF-8-инструментах —
это ожидаемо, а не повреждение.

:::danger Главная причина порчи файлов в проекте
Если файл с кириллицей сохранить как UTF-8, in-game текст ломается в «кракозябры» — например,
искажённые диалоги смартфона и строки на экране вместо читаемого русского.
:::

## Почему UTF-8 ломает cp1251-файлы необратимо

Инструменты, которые перезаписывают файл целиком как UTF-8 (обычные редакторы «сохранить»),
читают cp1251-файл как UTF-8: каждый байт кириллицы, не являющийся валидным UTF-8, превращается
в `U+FFFD` (`EF BF BD`). Это **невосстановимо** — оригинальные буквы теряются.

## Практические правила

- **Не** перекодируйте файлы целиком и не меняйте кодировку ради мелкой правки. Сохраняйте текущую
  кодировку файла при редактировании.
- **Никогда** не редактируйте cp1251-файл с кириллицей инструментами, которые пересохраняют весь файл
  как UTF-8 — даже для ASCII-only изменения. Вместо этого используйте **байт-безопасные** инструменты:
  - `perl -i -pe 's/.../.../'` (работает с сырыми байтами);
  - `sed -i` по номерам строк;
  - PowerShell через `[byte[]]` (`[System.IO.File]::ReadAllBytes/WriteAllBytes`).
- Это касается **и комментариев** — `// пусто`, сохранённый как UTF-8, тоже портит кодировку файла.
  Если не можете написать комментарий в cp1251 — пишите его на английском (ASCII).

## Если файл уже испорчен

Кириллица, превращённая в `EF BF BD` (рендерится как `пїЅ` при чтении в cp1251), не чинится на месте.
Восстановите чистый текст и заново примените только ASCII-правки байт-безопасными инструментами:

```bash
git checkout HEAD -- <file>
```

## Как добавлять русский текст

Пишите такой файл (или фрагмент) явно в cp1251 — например, через PowerShell:

```powershell
[System.Text.Encoding]::GetEncoding(1251)
[System.IO.File]::WriteAllText(...)
```

Проверяйте байты: `января` → `FF ED E2 E0 F0 FF`. Пример cp1251-файла с русскими названиями
месяцев/дней недели — `sources/player/phone_clock.inc`.

## Проверка перед завершением правок

Корректный cp1251-файл не содержит лид-байтов `D0`/`D1`, за которыми идёт байт-продолжение `0x80–0xBF`.
Проверка:

```bash
grep -laP '[\xD0\xD1][\x80-\xBF]' sources/**/*.inc sources/**/*.pwn
```

Команда должна ничего не вывести. Если файл помечен — сделайте байтовую замену UTF-8-последовательности
на cp1251-эквивалент (не пересохраняя весь файл как UTF-8).
